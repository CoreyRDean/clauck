# User Stories

Each file in this directory describes a user's intent — what they want to accomplish, why, and what success looks like. Stories are not feature specs or implementation proposals. They describe the problem, not the solution.

## How to read these

Each story follows the same structure:
- **Who** — the type of user (casual, power user, team lead, etc.)
- **Intent** — what they want, in their words
- **Context** — why this matters to them, what they've tried
- **Success** — what the world looks like when it works

## How agents use these

When executing user intent, agents can reference these stories to understand the space of possibilities and find precedent for how similar intents have been satisfied. Stories ground suggestions in real use cases rather than abstract feature descriptions.

## How to contribute a story

Write a new Markdown file in this directory describing a workflow you want supported. Focus on the problem and the desired outcome. Don't prescribe the solution — let the system's capabilities determine how to satisfy the intent. Open a PR with the story; discussion happens in the PR.
